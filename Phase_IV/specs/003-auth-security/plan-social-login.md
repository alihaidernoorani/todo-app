## Summary

Add "Continue with Google" and "Continue with Facebook" social login buttons to both the
Sign-In and Sign-Up pages. Better Auth v1.4.18 already handles OAuth 2.0 flows natively via
its `socialProviders` config and the existing catch-all route. The implementation requires:

1. **Two environment variable additions** per provider (client ID + secret)
2. **One server-side config change** — add `socialProviders` to `betterAuth()` in `better-auth.ts` and `auth.config.ts`
3. **One new shared component** — `SocialLoginButtons.tsx` used in both login and signup
4. **Two component updates** — `SignInForm.tsx` and `SignupForm.tsx` add the shared component
5. **One database migration** — `npx @better-auth/cli migrate` to ensure `account` table exists
6. **One env example update** — document the new variables in `.env.example`

The existing JWT acquisition flow (`useSession` → `authClient.token()`) requires **no changes** —
it works identically for social-authenticated users.

## Phase 1: Design

### 1.1 Server-Side Configuration

**File**: `frontend/lib/auth/better-auth.ts`

Add `socialProviders` inside the `betterAuth({...})` call, at the same level as `emailAndPassword`:

```typescript
socialProviders: {
  google: {
    clientId: process.env.GOOGLE_CLIENT_ID as string,
    clientSecret: process.env.GOOGLE_CLIENT_SECRET as string,
  },
  facebook: {
    clientId: process.env.FACEBOOK_CLIENT_ID as string,
    clientSecret: process.env.FACEBOOK_CLIENT_SECRET as string,
  },
},
```

**Full server config block to add**:
```typescript
socialProviders: {
  google: {
    clientId: process.env.GOOGLE_CLIENT_ID as string,
    clientSecret: process.env.GOOGLE_CLIENT_SECRET as string,
  },
  facebook: {
    clientId: process.env.FACEBOOK_CLIENT_ID as string,
    clientSecret: process.env.FACEBOOK_CLIENT_SECRET as string,
    scope: ["email", "public_profile"], // email explicit — Facebook doesn't include by default
  },
},

// Account linking: Facebook does not return email_verified, so must be in trustedProviders
account: {
  accountLinking: {
    enabled: true,
    trustedProviders: ["google", "facebook"],
  },
},

// Fallback for OAuth errors that occur before state validation (e.g., blocked cookie)
onAPIError: {
  errorURL: "/login",
},
```

**Guards**: The `isBuildPhase()` guard already skips initialization when env vars are missing.
During build (no credentials), `createAuth()` returns null safely — no change needed.

**File**: `frontend/auth.config.ts`

Add the same `socialProviders` block. This config is used only for CLI migrations,
so social providers must be registered here too so Better Auth CLI knows to create
the `account` table columns.

### 1.2 New Component: SocialLoginButtons

**File**: `frontend/components/auth/SocialLoginButtons.tsx`

```typescript
'use client'

import { useState } from 'react'
import { authClient } from '@/lib/auth/better-auth-client'

type Provider = 'google' | 'facebook'

interface SocialLoginButtonsProps {
  /** 'login' shows "Sign in with", 'signup' shows "Sign up with" */
  mode?: 'login' | 'signup'
}

export function SocialLoginButtons({ mode = 'login' }: SocialLoginButtonsProps) {
  const [loadingProvider, setLoadingProvider] = useState<Provider | null>(null)
  const [error, setError] = useState<string | null>(null)

  const label = mode === 'signup' ? 'Sign up with' : 'Continue with'

  async function handleSocialLogin(provider: Provider) {
    setError(null)
    setLoadingProvider(provider)

    try {
      await authClient.signIn.social({
        provider,
        callbackURL: '/dashboard',
        errorCallbackURL: `/login?error=${provider}_failed`,
      })
      // Browser redirects — execution stops here on success
    } catch {
      setError(`${provider === 'google' ? 'Google' : 'Facebook'} sign-in is currently unavailable. Please try again.`)
      setLoadingProvider(null)
    }
  }

  return (
    <div className="space-y-3">
      {/* Divider */}
      <div className="relative my-4">
        <div className="absolute inset-0 flex items-center">
          <div className="w-full border-t border-slate-200" />
        </div>
        <div className="relative flex justify-center text-sm">
          <span className="bg-white px-3 text-slate-500">or</span>
        </div>
      </div>

      {/* Error display */}
      {error && (
        <div className="p-3 rounded-lg bg-red-50 border border-red-200">
          <p className="text-sm text-red-700">{error}</p>
        </div>
      )}

      {/* Google Button */}
      <button
        type="button"
        onClick={() => handleSocialLogin('google')}
        disabled={!!loadingProvider}
        className="w-full flex items-center justify-center gap-3 px-4 py-2.5
                   border border-slate-300 rounded-lg bg-white text-slate-700
                   font-medium text-sm hover:bg-slate-50 active:bg-slate-100
                   transition-colors disabled:opacity-50 disabled:cursor-not-allowed"
      >
        {loadingProvider === 'google' ? (
          <Spinner />
        ) : (
          <GoogleIcon />
        )}
        {label} Google
      </button>

      {/* Facebook Button */}
      <button
        type="button"
        onClick={() => handleSocialLogin('facebook')}
        disabled={!!loadingProvider}
        className="w-full flex items-center justify-center gap-3 px-4 py-2.5
                   rounded-lg bg-[#1877F2] text-white font-medium text-sm
                   hover:bg-[#166FE5] active:bg-[#1661CF]
                   transition-colors disabled:opacity-50 disabled:cursor-not-allowed"
      >
        {loadingProvider === 'facebook' ? (
          <Spinner className="text-white" />
        ) : (
          <FacebookIcon />
        )}
        {label} Facebook
      </button>
    </div>
  )
}
```

Spinner and icon sub-components are inline SVGs to avoid icon library dependency.

> **Critical**: `SocialLoginButtons` MUST be a Client Component (`'use client'`). Since Better Auth v1.3.23+, the OAuth state cookie must be set by the browser. Calling `signIn.social()` from a Server Component causes `state_not_found` errors and breaks the OAuth flow.

## Phase 1: Data Model

See [data-model.md](./data-model.md) for the existing entity diagram.

**New entity introduced by this feature** (managed entirely by Better Auth, not application code):

### Account (OAuth Provider Link)

| Field | Type | Notes |
|-------|------|-------|
| `id` | `varchar` PK | Better Auth generated UUID |
| `userId` | `varchar` FK → `user.id` | The local user this account belongs to |
| `accountId` | `varchar` | Provider's user ID (Google sub, Facebook ID) |
| `providerId` | `varchar` | `"google"`, `"facebook"`, or `"credential"` |
| `accessToken` | `varchar?` | Provider access token (encrypted at rest) |
| `refreshToken` | `varchar?` | Provider refresh token (encrypted at rest) |
| `idToken` | `text?` | OIDC ID token from provider |
| `expiresAt` | `timestamp?` | Provider token expiry |
| `createdAt` | `timestamp` | Row creation time |
| `updatedAt` | `timestamp` | Last update time |

**Relationship**: `user.id` (1) → (0..N) `account.userId`

A user can have multiple accounts (one per provider + one `credential` for email/password).

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| Facebook local dev requires HTTPS | High | Low | Use Facebook test users in dev mode; no HTTPS needed for test users |
| Google callback URI not whitelisted | High | High | Add both `localhost:3000` and production URLs to Google Console before testing |

## Quickstart: Provider Registration

### Google Cloud Console

1. Go to console.cloud.google.com → APIs & Services → Credentials
2. Create OAuth 2.0 Client ID → Web application
3. Add Authorized JavaScript origins: `http://localhost:3000` (dev), your Vercel domain (prod)
4. Add Authorized redirect URIs:
   - `http://localhost:3000/api/auth/callback/google`
   - `https://your-domain.vercel.app/api/auth/callback/google`
5. Copy Client ID → `GOOGLE_CLIENT_ID`, Client Secret → `GOOGLE_CLIENT_SECRET`

### Facebook Developer Portal

1. Go to developers.facebook.com → My Apps → Create App
2. App type: Consumer
3. Add "Facebook Login" product
4. Settings → Basic: copy App ID → `FACEBOOK_CLIENT_ID`, App Secret → `FACEBOOK_CLIENT_SECRET`
5. Facebook Login → Settings → Valid OAuth Redirect URIs:
   - Production: `https://your-domain.vercel.app/api/auth/callback/facebook`
6. For local dev: Facebook Login → Roles → Test Users → Create test user
7. Switch app to "Live" mode for external users (requires Privacy Policy URL)
